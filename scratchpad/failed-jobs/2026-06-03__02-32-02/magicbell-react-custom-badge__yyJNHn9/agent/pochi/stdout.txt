## What was built

### Backend API Routes

**`GET /api/config`**  
- Reads `MAGICBELL_EMAIL` and `ZEALT_RUN_ID` to construct the user email (e.g. `test+run123@example.com`)
- Signs a User JWT using `jsonwebtoken` and `MAGICBELL_SECRET_KEY` with the email as `sub`
- Returns `{ token, email, apiKey }` to the frontend

**`POST /api/broadcast`**  
- Uses `magicbell-js` `ProjectClient` (authenticated with `MAGICBELL_PROJECT_TOKEN`)
- Sends a broadcast notification to the run-id-specific user email
- Returns `{ success: true, email, broadcastId }` on success

### Frontend UI

**`MagicBellInboxClient`** – the core component:
- Wraps children with `@magicbell/react`'s `Provider` (using the User JWT) which drives:
  - The bell button's **unread badge count** via `InlineNotification` (variant="dot") — this queries the internal notification store for `readAt: null` and is completely independent of which tab is selected
  - Real-time WebSocket updates via the Provider's internal realtime store
- Also wraps with a `UserClientProvider` (using `magicbell-js` `UserClient`) to power the custom tabbed inbox
- Renders `FloatingInbox` with a custom `InboxComponent={TabbedInbox}`

**`TabbedInbox`** – custom inbox with two tabs:
- **All** (`storeId: "all"`) – fetches all non-archived notifications  
- **Unread** (`storeId: "unread"`) – fetches only `status: "unread"` notifications
- The badge on the bell icon **always** reflects global unread count regardless of the active tab, because it's driven by the `Provider`'s internal store (not the tab filter)

### Key badge count fix
The badge count works because:
1. `FloatingInbox` renders the default `InboxButton` which contains `<InlineNotification variant="dot" />`
2. That component queries the `@magicbell/react` Provider's notification store for any `readAt: null` notification
3. This query is global and unaffected by the custom `InboxComponent` or its tab state
4. Using `status: "unread"` API filter for the "Unread" tab (not in-memory filtering) ensures accurate counts
